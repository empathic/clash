## Workstream 1: Starlark Policy Mutation (#432 + #431)

### Problem

The CLI commands `clash policy allow` and `clash policy deny` only work with `.json` policy files. Users who write policies in Starlark (the recommended format) must manually edit their `.star` files. Additionally, `clash policy edit --raw` does not validate the file before replacing the original, risking broken configs.

### Design

#### 1A: `policy allow`/`deny` for .star files (#432)

**Approach:** Managed section with codegen. When the target policy is a `.star` file, the mutation pipeline appends rules to a designated section and references them in the `policy()` call.

**Mutation flow for `.star` files:**

1. **Parse** the existing `.star` file into the codegen AST (`Vec<Stmt>`) using the existing tree-sitter ‚Üí AST infrastructure in `clash_starlark/src/codegen/`
2. **Find or create the managed section** ‚Äî scan for the comment marker `# clash-managed rules`. If absent, append it before the final `policy()` call
3. **Generate the new rule** using existing codegen builders (`node_json_to_expr()`, `builder::allow()`, etc.) ‚Äî the same ones the `convert` command uses
4. **Append as a variable assignment** in the managed section:
   ```starlark
   # clash-managed rules
   _clash_rule_0 = when({"Bash": {"git": {"push": allow()}}})
   ```
5. **Update the `policy()` call** using `codegen::mutate::policy_rules_mut()` to append the new variable as an `Ident` reference to the rules list
6. **Validate** by running the result through `clash_starlark::evaluate()`. If validation fails, abort and show the error ‚Äî do not write the file
7. **Show a tree diff** of the compiled policy before and after (see Tree Diff section below), and ask for confirmation before writing
8. **Serialize and write** via `codegen::serialize()`

**Duplicate detection:** Before inserting, compile both the existing policy and the candidate rule to JSON IR. Use the existing `same_match_chain()` logic from `manifest_edit.rs` to check if a rule with the same observable chain already exists. If so, replace the existing managed variable's value rather than appending a new one.

**Variable naming:** `_clash_rule_N` where N increments based on existing managed rules. The underscore prefix signals these are generated, not user-authored.

**Entry points unchanged:** The `--broad` flag, hash-based lookup, `--tool`, and `--bin` flags all produce a `Node` as before. The only change is that `apply_mutation()` branches on file extension ‚Äî `.json` uses the existing `manifest_edit` path, `.star` uses the new codegen path.

#### 1B: `policy edit --raw` validation (#431)

**Approach:** visudo-style tempfile editing with validation and tree diff confirmation.

**Flow:**

1. **Copy** the policy file to a tempfile in the same directory (so relative `load()` paths resolve correctly)
2. **Open** the tempfile in `$EDITOR`
3. **On editor exit, validate:**
   - `.star` files: run through `clash_starlark::evaluate()`
   - `.json` files: JSON parse + schema validation
4. **If invalid:** display the error and prompt: `[e]dit again, [a]bort?`
   - `e` ‚Üí reopen editor on the tempfile (preserving changes)
   - `a` ‚Üí delete tempfile, exit with no changes
5. **If valid:** show a tree diff (see below) of the compiled policy before vs after
6. **Prompt:** `Apply these changes? [y/n]`
   - `y` ‚Üí replace original file with tempfile contents
   - `n` ‚Üí delete tempfile, exit with no changes

This applies to both `.star` and `.json` policy files.

#### Tree Diff Utility

A shared utility for displaying before/after policy changes, used by both `policy allow` (on `.star` files) and `policy edit --raw`.

**Implementation:**

1. Compile both old and new policy sources to `CompiledPolicy`
2. Render each with the existing `format_tree()` from `policy/format.rs` ‚Äî the same renderer used by `clash status`
3. Produce a unified diff of the two `Vec<String>` outputs using a Myers diff algorithm
4. Display with `+`/`-` prefixes, colored with `style::green()` / `style::red()`

**Example output:**
```
Policy changes:
 ‚îú‚îÄ‚îÄ Bash
-‚îÇ   ‚îî‚îÄ‚îÄ git push ‚Üí deny
+‚îÇ   ‚îú‚îÄ‚îÄ git push ‚Üí allow [sandbox: git_full]
+‚îÇ   ‚îî‚îÄ‚îÄ git pull ‚Üí allow [sandbox: git_safe]
 ‚îî‚îÄ‚îÄ Read ‚Üí allow
```

**Location:** New module `clash/src/policy/diff.rs` with a public function:
```rust
pub fn format_tree_diff(before: &CompiledPolicy, after: &CompiledPolicy) -> Vec<String>
```

### Key Files

| File | Changes |
|------|---------|
| `clash/src/cmd/policy.rs` | Branch `apply_mutation()` on file extension; rewrite `handle_edit` for `--raw` with validation loop |

## Workstream 2: gh CLI in Git Sandbox (#430)

### Problem

The prebuilt `git_safe` and `git_full` sandboxes don't include filesystem access for the GitHub CLI (`gh`), so `gh` commands fail inside these sandboxes.

### Design

Add read access to `~/.config/gh/**` in both sandbox definitions in `clash_starlark/stdlib/sandboxes.star`:

```starlark
# In both git_safe and git_full fs rules, under "$HOME":
glob(".config/gh/**"): allow("r"),
```

**What `gh` needs:**
- `~/.config/gh/config.yml` ‚Äî preferences and aliases
- `~/.config/gh/hosts.yml` ‚Äî host configs and auth tokens

**What's already covered by the git sandboxes:**
- Network access (HTTPS + SSH) ‚Äî required for GitHub API calls
- `~/.ssh/**` ‚Äî SSH authentication
- `~/.gitconfig` and `~/.config/git/**` ‚Äî git interop
- `$TMPDIR/**` ‚Äî temp files

Read-only access is sufficient. `gh` does not need to write its config during normal operations (`gh pr create`, `gh issue list`, etc.).

### Key Files

| File | Changes |
|------|---------|
| `clash_starlark/stdlib/sandboxes.star` | Add `glob(".config/gh/**"): allow("r")` to `git_safe` and `git_full` |

## Workstream 3: Runtime Settings Import (#427)

### Problem

Currently, importing Claude Code settings into Clash is a one-time operation (`clash init` / `cmd/import_settings.rs`). If the user changes their Claude Code permissions, Clash doesn't pick them up. Users want a way to dynamically pull Claude Code settings into their Starlark policy on every evaluation.

### Design

#### Native Starlark function: `from_claude_settings()`

**User-facing API:**

```starlark
load("@clash//claude_compat.star", "from_claude_settings")

policy("main",
    default = ask(),
    rules = [
        # user's custom rules take priority (listed first)
        when({"Bash": {"git": allow()}}),
    ] + from_claude_settings(user=True, project=True),
)
```

**Parameters:**
- `user` (bool, default `True`) ‚Äî include rules from `~/.claude/settings.json`
- `project` (bool, default `True`) ‚Äî include rules from project-level `.claude/settings.json`
- `session` (bool, default `False`) ‚Äî include rules from session-level settings

**Returns:** A `list[MatchTreeNode]` ‚Äî the same type that `when()` returns, composable with `+` in the `policy()` rules list.

#### Implementation

1. **Native Rust function** registered via `#[starlark_module]` in `clash_starlark/src/globals.rs`
   - Calls `claude_settings::ClaudeSettings` to load and parse the requested layers
   - Converts permission entries to match tree nodes

2. **Shared classification logic** ‚Äî extract the core of `import_settings.rs`'s permission classification into a shared module (e.g., `clash/src/policy/settings_compat.rs` or within `claude_settings` crate):
   - `classify_permission()` ‚Äî parse permission strings like `"Bash(git:*)"`, `"Read(.env)"`, `"Read(src/**/*.ts)"`
   - Permission ‚Üí Node conversion (tool rules, bash prefix rules, file access rules)
   - Both the one-time import CLI and the runtime function use this shared code

3. **Glob pattern support** ‚Äî unlike the current one-time import which skips glob patterns, `from_claude_settings()` converts them to Clash's `glob()` path matcher. For example, `"Read(src/**/*.ts)"` becomes a rule matching `Read` with a `glob("src/**/*.ts")` file path condition. This fix also applies to the one-time import.

4. **Stdlib module** ‚Äî `clash_starlark/stdlib/claude_compat.star` re-exports the native function:
   ```starlark
   # claude_compat.star
   # from_claude_settings() is a native function.
   # Load it to dynamically import Claude Code permission settings as Clash rules.
   #
   # Usage:
   #   load("@clash//claude_compat.star", "from_claude_settings")
   #   policy("main", rules = [...] + from_claude_settings())
   ```
   The native function is injected during module loading, similar to how `std.star` pre-injects `allow()`, `deny()`, etc.

5. **Skipped entries:** MCP tools are skipped (Clash doesn't manage MCP). No warning is emitted since this runs silently at eval time.

6. **Performance:** Settings are read fresh on each Starlark evaluation (once per hook invocation). The `claude_settings` read is a few small JSON files ‚Äî no caching needed.

#### Dependency

`clash_starlark` currently does not depend on `claude_settings`. This change adds that dependency. Since both are workspace crates, this is straightforward ‚Äî add `claude_settings` to `clash_starlark/Cargo.toml`.

### Key Files

| File | Changes |
|------|---------|
| `clash_starlark/src/globals.rs` | Register `from_claude_settings()` native function |
| `clash_starlark/src/loader.rs` | Register `claude_compat.star` as loadable stdlib module |
| `clash_starlark/stdlib/claude_compat.star` | **New** ‚Äî stdlib module re-exporting the native function |
| `clash_starlark/Cargo.toml` | Add `claude_settings` dependency |
| `clash/src/cmd/import_settings.rs` | Extract shared classification logic out |
| `clash/src/policy/settings_compat.rs` (or `claude_settings/src/compat.rs`) | **New** ‚Äî shared permission classification and conversion |

## Testing Strategy

### Workstream 1 (Starlark Mutation)
- **Unit tests** for managed section parsing, creation, and variable insertion in codegen
- **Unit tests** for tree diff output
- **Integration tests** via clester: `clash policy allow "git push"` on a `.star` policy file, verify the file is valid and the rule appears in `clash policy show`
- **Integration tests** for `policy edit --raw`: simulate editor writes with valid/invalid content, verify validation loop behavior

### Workstream 2 (gh Sandbox)
- **Integration test** via clester: run `gh --version` inside a `git_safe` sandbox, verify it succeeds
- **Integration test**: verify `gh auth status` can read config (or fails gracefully with auth error, not permission error)

### Workstream 3 (Runtime Settings Import)
- **Unit tests** for the native function: mock Claude Code settings, verify correct rule output
- **Unit tests** for glob pattern conversion
- **Integration tests** via clester: create a `.star` policy using `from_claude_settings()`, set up Claude Code settings with known permissions, verify the compiled policy includes them
- **Regression tests**: ensure MCP tools are still skipped, ensure empty settings produce empty rule list
