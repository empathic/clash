# Clash Policy Language Specification

**Version:** 2
**Status:** Draft

---

## Design Context

Policies are primarily generated by interactive tooling (TUI). The s-expression format serves as a readable, versionable, inspectable artifact. Expert users may hand-author policies and validate them with `clash policy validate`.

---

## Overview

Clash is a policy language for defining permission and sandboxing rules for AI agent tool invocations. Policies are expressed as s-expressions that form a strict decision tree. The runtime evaluates this tree to produce one of two outcomes:

1. **A terminal decision** — `:allow`, `:deny`, or `:ask`
2. **A set of runtime constraints** — restrictions on filesystem, network, and other resources, derived from the surviving decision tree

The language is agnostic to enforcement mechanism. The same policy tree governs both agent-harness tool invocations and raw command execution. Runtime constraints (e.g., sandbox filesystem and network restrictions) are not expressed as separate forms — they are derived from the decision tree. The `ctx` dimensions that appear in the surviving subtree define what the invocation may access; the runtime is responsible for mapping these decisions to the appropriate enforcement layer (tool argument validation, OS-level sandboxing, etc.).

---

## Document Structure

A policy file is an ordered sequence of top-level forms:

```scheme
(version 2)
(use <policy-name>)

(def <name> <value>)
...

(policy <name> <body...>)
...
```

### `version`

Required. Must be the first form. Declares the schema version, which determines the valid shape of the `ctx` type tree.

```scheme
(version 2)
```

### `use`

Required. Declares the entry point policy — the root of the decision tree. Must reference a `policy` defined in the file.

```scheme
(use "main")
```

### `def`

Binds a name to a value. Definitions are global and may be referenced by name anywhere after their declaration. Values may be any valid expression: literals, lists, or compound forms including `match` blocks.

```scheme
(def tmpdirs ["/tmp" "/var/folders" "/var/private/folders"])

(def github-net
  (match ctx.http.domain
    (default :deny)
    "github.com" :allow))
```

Names must match `[a-zA-Z][a-zA-Z0-9\-]*`.

### `policy`

A named node in the decision tree. Policies are the primary unit of composition.

```scheme
(policy <name:string>
  <body...>)
```

The body may contain any combination of: `default`, `when`, `match`, `include`, and named `def` references.

## Core Forms

### Terminal Symbols

Three terminal symbols represent final decisions:

| Symbol | Meaning | Context |
|--------|---------|---------|
| `:allow` | Permit the operation | Tools, commands |
| `:deny` | Reject the operation | Tools, commands |
| `:ask` | Prompt the user for a runtime decision | Tools only |

`:ask` is only valid in tool invocation contexts. Applying `:ask` to a raw command is a validation error.

### `default`

Sets the fallthrough decision for unmatched cases within the enclosing scope. May appear at any level in the tree: inside a `policy`, inside a `match`, or alongside constraints.

```scheme
(default :deny)
```

Resolution: when evaluating a decision, the nearest ancestor `default` wins for unmatched cases — the runtime walks up the tree from the unmatched node until it finds a `default` in an enclosing `policy`, `when`, or `match`. If no `default` is found in any ancestor scope, the result is a validation error (policies must be total).

### `when`

A guard. Contains a single predicate and one or more body expressions. The body expressions evaluate only when the predicate holds.

```
(when <predicate> <body...>)
```

The predicate is a single boolean expression. The body is one or more expressions that are all evaluated when the guard passes.

```scheme
(when (eq ctx.process.command "cargo")
  (match ctx.http.domain
    (or "github.com" "crates.io") :allow)
  (match ctx.fs.path
    (subpath (joinpath $PWD "target")) :allow))
```

`when` is a **guard**: one predicate, multiple effects. It does not branch — it gates.

### `match`

Dispatches on a value. Contains a subject expression and one or more pattern/result arms. The first matching arm wins.

```
(match <subject>
  <pattern> <result>
  <pattern> <result>
  ...)
```

The subject may be a single value or a tuple of values (destructured matching):

```scheme
;; Single value dispatch
(match ctx.http.domain
  "github.com" :allow
  "crates.io" :allow)

;; Tuple dispatch
(match [ctx.fs.action ctx.fs.path]
  ["read" (subpath $PWD)] :allow
  [* (subpath (joinpath $PWD "target"))] :allow)
```

`match` is **dispatch**: one subject, multiple branches. It does not gate — it routes.

A `(default ...)` may appear inside a `match` to set the fallthrough for unmatched arms:

```scheme
(match ctx.http.domain
  (default :deny)
  "github.com" :allow)
```

### `include`

Incorporates a named policy into the current policy's tree. Included policies are evaluated as part of the enclosing policy.

```scheme
(policy "main"
  (default :deny)
  (include "rust")
  (include "web"))
```

Circular includes are invalid. The policy graph must form a strict tree.

## Patterns and Expressions

### Literals

- **Strings:** `"github.com"`, `"read"`
- **Symbols:** `:allow`, `:deny`, `:ask`
- **Wildcard:** `*` — matches any value
- **Regex:** `/pattern/` or `/pattern/(flags)` — matches against the string representation of the subject

```scheme
(match ctx.http.domain
  /.*\.github\.com/ :allow
  /.*\.internal\.corp/(i) :deny)
```

### Lists

Ordered collections enclosed in brackets:

```scheme
["/tmp" "/var/folders" "/var/private/folders"]
```

### Tuples (Destructuring)

Used in `match` subjects and patterns to match against multiple values simultaneously:

```scheme
(match [ctx.fs.action ctx.fs.path]
  ["read" (subpath $PWD)] :allow
  ["write" /.*\.log/] :allow
  [* *] :deny)
```

### Built-in Functions

#### `or`

Matches if any of the arguments match. Used in both predicates and patterns.

```scheme
(or "github.com" "crates.io")
(or "cargo" "rustc" "rustup")
```

#### `not`

Negation. Matches if the argument does not match.

```scheme
(not "github.com")
(not (subpath "/etc"))
```

#### `eq`

Equality test. Used in `when` predicates.

```scheme
(when (eq ctx.process.command "cargo") ...)
```

#### `subpath`

Path prefix match. Returns true if the subject is a subpath of the argument.

```scheme
(subpath $PWD)                         ;; anything under $PWD
(subpath (joinpath $PWD "target"))     ;; anything under $PWD/target
(subpath ["/tmp" "/var/folders"])       ;; anything under /tmp or /var/folders
```

#### `joinpath`

Joins path segments.

```scheme
(joinpath $PWD "target")    ;; $PWD/target
(joinpath $HOME ".config")  ;; $HOME/.config
```

## Invocation Types

The `when` guard and `match` dispatch can test what kind of invocation is being evaluated. Invocation type predicates are **pattern matches** against the invocation kind and its primary identifier (e.g., command name, tool name). They are syntactic sugar — `(command "cargo")` matches the same input as `ctx.process.command`, but scoped to only activate for command invocations. Use invocation types to guard a subtree by invocation kind; use `ctx` fields for fine-grained dispatch within that subtree.

### `command`

Matches raw command execution (e.g., shell commands). Commands run outside the agent harness and are subject to sandbox enforcement.

```scheme
(when (command (or "cargo" "rustc")) ...)
(when (command /npm.*/) ...)
```

### `tool`

Matches agent-harness tool invocations. Tools run within the agent harness and are subject to tool-level permission checks.

```scheme
(when (tool "WebFetch") ...)
(when (tool (or "WebSearch" "WebFetch")) ...)
```

### `mcp`

Matches MCP (Model Context Protocol) tool invocations. Can match on server name, tool name, or both.

```scheme
(when (mcp "puppeteer") ...)
(when (mcp /puppeteer.*/) ...)

(match ctx.mcp.tool
  "puppeteer_navigate" :allow
  "puppeteer_screenshot" :ask)
```

### `agent`

Matches subagent spawning.

```scheme
(when (agent "Explore") :allow)
(when (agent (or "Explore" "Verify")) :ask)
```

## Runtime Context (`ctx`)

`ctx` is a well-typed, nested structure representing the runtime context of an invocation. The shape of `ctx` is fixed per `version`. Referencing an invalid `ctx` path is a parse-time validation error.

### Version 2 Schema

```
ctx
├── http
│   ├── domain        : string   — destination domain
│   ├── method        : string   — HTTP method (GET, POST, etc.)
│   ├── port          : int      — destination port
│   └── path          : string   — URL path
├── fs
│   ├── action        : string   — operation type (read | write | edit | glob | grep)
│   ├── path          : string   — file path
│   └── exists        : bool     — whether the target exists
├── process
│   ├── command       : string   — executable name
│   └── args          : [string] — argument list
├── tool
│   ├── name          : string   — tool name
│   └── args          : {string?} — tool arguments (dynamic, nullable accessors)
├── mcp
│   ├── server        : string   — MCP server name
│   └── tool          : string   — MCP tool name
├── agent
│   └── name          : string   — subagent name
└── state             : string   — agent state (idle, running, etc.)
```

### Environment Variables

Environment variables are referenced with `$` prefix and are expanded at evaluation time:

```scheme
$PWD      ;; current working directory
$HOME     ;; user home directory
```

### Nullable Accessors

Fields in dynamic subtrees (e.g., `ctx.tool.args`) use the `?` suffix to indicate that the field may not exist for all invocations. There is no null value in the language. When a nullable accessor references an absent field, the **entire enclosing `match` is short-circuited** — no arms are evaluated, and the nearest ancestor `default` applies.

```scheme
;; If ctx.tool.args.file_path? is absent, this entire match is skipped
;; and the nearest ancestor default applies
(match ctx.tool.args.file_path?
  (subpath $PWD) :allow
  * :deny)              ;; * only matches when the field is present but no other arm matched
```

Referencing a dynamic field *without* `?` is a validation error — the author must explicitly acknowledge that the field may be absent.

### Extension Point: `ctx.mcp.<server>`

In future versions, MCP servers may register custom subtrees under `ctx.mcp.<server>.*`. The core `ctx` namespaces (`http`, `fs`, `process`, `tool`, `agent`, `state`) remain fixed per version. Only `ctx.mcp.*` is the designated extension point for third-party type definitions.

## Evaluation Model

### Tree Shaking

Policy evaluation is a tree walk. The runtime traverses the policy tree, pruning branches based on guards (`when`) and dispatching through branches (`match`). Whatever subtree survives defines the constraints for the invocation.

### Decision Resolution

1. Walk the tree from the entry policy.
2. Evaluate `when` guards — if the predicate fails, prune the entire subtree.
3. Evaluate `match` arms — the first matching arm wins for that `match`.
4. Collect all constraints from the surviving subtree.
5. For any unmatched dimension, walk up the tree from the unmatched node and apply the nearest ancestor `default`.
6. If no `default` exists for an unmatched dimension, the policy is invalid (parse-time error).

### Worked Example: Evaluation Trace

Evaluation has two phases: **decision** (walk the tree with known inputs, prune branches) and **constraint derivation** (the surviving subtree defines what the runtime enforces).

Given the "rust" policy from the complete example, evaluate `cargo build`:

**Known inputs (decision phase):**
```
ctx.process.command = "cargo"
ctx.process.args    = ["build"]
$PWD                = "/home/user/project"
```

**Phase 1 — Decision (tree shaking):**

1. Enter `"main"` (via `(use "main")`). Default is `:deny`.
2. Evaluate `(include "rust")` — enter `"rust"`. Default is `:deny`.
3. Evaluate `(when (command (or "cargo" "rustc")) ...)`:
   - Pattern `(or "cargo" "rustc")` matched against `"cargo"` → **matches**. Enter body.
4. The `when` body survives. Decision: **`:allow`** (the command is permitted).

**Phase 2 — Constraint derivation:**

The surviving subtree contains two `match` blocks. These are not evaluated against inputs — they become runtime constraints:

- `(match ctx.http.domain ...)` → **HTTP constraint**: allow connections to `github.com` and `crates.io`, deny all others.
- `(match [ctx.fs.action ctx.fs.path] ...)` → **Filesystem constraint**: allow reads under `$PWD`, allow any operation under `$PWD/target` and `tmpdirs`, deny all others.

The runtime maps these constraints to enforcement mechanisms. For example: start an HTTP proxy outside the sandbox that enforces the domain allowlist, restrict the sandbox to localhost-only network access on the proxy port, and apply filesystem restrictions via OS-level sandboxing.

**Contrast — unmatched command:**

If `ctx.process.command = "node"`, the `when` guard in step 3 fails. The body is pruned. No other `include` in `"main"` matches. The nearest ancestor `default` (`:deny`) applies. Decision: **`:deny`**, no constraints derived (the command is blocked entirely).

### Isolation

Each policy's constraint set is **isolated**. There is no merging or inheritance of constraints between policies. Reuse is achieved through `def` bindings, not through sandbox composition. The validator enforces this by rejecting policies where sibling forms could produce overlapping decisions for the same input (see validation rule 10) — so constraint collection is enumeration, never merging.

```scheme
;; Reuse through def — explicit, no merging ambiguity
(def rust-net
  (match ctx.http.domain
    (default :deny)
    (or "github.com" "crates.io") :allow))

(policy "rust"
  (default :deny)
  (when (command (or "cargo" "rustc"))
    rust-net
    (match [ctx.fs.action ctx.fs.path]
      ["read" (subpath $PWD)] :allow
      [* (subpath (joinpath $PWD "target"))] :allow)))
```

## Complete Example

```scheme
(version 2)
(use "main")

;; Shared definitions
(def tmpdirs ["/tmp" "/var/folders" "/var/private/folders"])

(def github-net
  (match ctx.http.domain
    (default :deny)
    "github.com" :allow))

;; Rust build tools
(policy "rust"
  (default :deny)
  (when (command (or "cargo" "rustc"))
    (match ctx.http.domain
      (default :deny)
      (or "github.com" "crates.io") :allow)
    (match [ctx.fs.action ctx.fs.path]
      ["read" (subpath $PWD)] :allow
      [* (subpath (joinpath $PWD "target"))] :allow
      [* (subpath tmpdirs)] :allow)))

;; Web tools
(policy "web"
  (default :deny)
  (when (tool (or "WebSearch" "WebFetch"))
    (match ctx.http.domain
      (default :deny)
      "github.com" :allow
      "wikipedia.org" :allow
      /.*\.anthropic\.com/ :allow)))

;; MCP tools
(policy "puppeteer"
  (default :deny)
  (when (mcp "puppeteer")
    (match ctx.mcp.tool
      "puppeteer_navigate" :ask
      "puppeteer_screenshot" :allow)))

;; Subagents
(policy "agents"
  (default :deny)
  (when (agent "Explore") :allow)
  (when (agent "Plan") :allow)
  (when (agent "Verify") :ask))

;; Entry point
(policy "main"
  (default :deny)
  (include "rust")
  (include "web")
  (include "puppeteer")
  (include "agents"))
```

## Grammar (EBNF)

```ebnf
comment       = ";" { any } newline

file          = version use { def | policy }
version       = "(" "version" integer ")"
use           = "(" "use" string ")"

def           = "(" "def" name expr ")"
policy        = "(" "policy" string body+ ")"

body          = default | when | match | include | terminal | name | expr

default       = "(" "default" terminal ")"
when          = "(" "when" predicate body+ ")"
match         = "(" "match" subject arm+ ")"
include       = "(" "include" string ")"

arm           = [ default ] pattern result
result        = terminal | body

predicate     = invocation | "(" "eq" expr expr ")"
invocation    = "(" invoc-type pattern ")"
invoc-type    = "command" | "tool" | "mcp" | "agent"

subject       = ctx-ref | "[" ctx-ref+ "]"
pattern       = literal | wildcard | regex | or-expr | not-expr | subpath-expr | "[" pattern+ "]"

expr          = literal | name | list | or-expr | subpath-expr | joinpath-expr | ctx-ref | env-var
literal       = string | symbol | integer
wildcard      = "*"
regex         = "/" chars "/" [ "(" flags ")" ]
symbol        = ":" name
string        = '"' chars '"'
name          = alpha { alpha | digit | "-" }
env-var       = "$" name
ctx-ref       = "ctx" "." name { "." name } [ "?" ]
list          = "[" expr* "]"

or-expr       = "(" "or" expr+ ")"
not-expr      = "(" "not" pattern ")"
subpath-expr  = "(" "subpath" expr ")"
joinpath-expr = "(" "joinpath" expr+ ")"

terminal      = ":allow" | ":deny" | ":ask"
```

## Validation Rules

1. `(version ...)` must be the first form in the file, followed by `(use ...)`.
2. `(use ...)` must reference a `policy` defined in the file.
3. All `ctx` references must be valid paths in the schema for the declared version.
4. Policy includes must form a strict tree (DAG with single parent per node). Circular references are invalid.
5. Every reachable leaf in the decision tree must resolve to a terminal symbol or be covered by a `default`.
6. `:ask` may only appear in contexts reachable from `tool`, `mcp`, or `agent` invocation types — not from `command`.
7. `def` names must be unique.
8. `policy` names must be unique.
9. All referenced `def` names and `include` policy names must be declared before use (forward references are invalid).
10. Regex patterns must be valid regular expressions. Flags are optional and follow the `/pattern/(flags)` syntax.
11. Sibling body forms within a `policy` or `when` block must govern non-overlapping `ctx` dimensions. If two sibling forms (e.g., two `when` guards or two `match` blocks) can both activate for the same input and produce decisions over the same `ctx` dimension, the policy is invalid. For any possible input, at most one decision may exist per dimension.

---

## Design Principles

- **Strict tree structure.** No cycles, no diamond inheritance, no merging ambiguity.
- **Isolation over inheritance.** Reuse through `def` bindings, not sandbox composition.
- **Enforcement agnostic.** Policies describe *what* to decide, not *how* to enforce. `ctx` dimensions are abstract types — the runtime maps them to enforcement mechanisms.
- **Total policies.** Every possible input must resolve to a decision. Uncovered cases are caught at parse time via `default` requirements.
- **Fixed schema, extensible via MCP.** Core `ctx` shape is versioned and static. MCP servers may extend `ctx.mcp.<server>.*` in future versions.
